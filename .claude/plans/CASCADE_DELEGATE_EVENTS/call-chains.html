The one idea: a cascade is a chain whose only fragile link is the delegate.Call right after a DB write. Every other link is robust and shared. The bug is always “the chain is severed before delegate.Call.”

3 The loop problem

Once these cascade, nothing stops A → B → A re-firing forever.

depth counter ✗ visited set ✗ origin flag ✗ Temporal dedup ✗ · unique IDs

Kahn's algorithm only checks inside one rule's DAG, at save time — structurally blind to cross-rule loops.

4 Every fix option, on the same chain

M1 & M2 restore/splice the missing link; the loop guard adds a gate at the shared chokepoint.

Handler.Execute▶ [write]▶ delegate.Call▶ DelegateHandler / OnEntityEvent▶ ProcessEvent▶ Temporal

M2 — restore the link

Cheap · local · un-breaks PR #176's workaround

  • add delegate.Call to alertbus.Create
  • … to approvalrequestbus.Create
  • … to CreateAllocationResult
  • wire a delegate field into alertbus
at: the bus write

M1 — splice a synthetic link

Architectural · generic actions have no bus

  • after the raw-SQL write, build a TriggerEvent
  • feed it straight into OnEntityEvent
  • also decide the validation gap
at: handler, post-write

Loop guard — add a gate

MANDATORY before shipping either fix

  • A. depth counter on the event — cap hops
  • B. visited-set — refuse re-entry
  • C. origin tag — suppress automated re-cascade (no flag today)
enforce once at: the chokepoint
