## Purpose

Customer-specific seed data pipeline. Lives entirely outside the Go codebase in
`deployments/customers/<name>/`. Provides real, meaningful data for a specific customer
deployment without touching the core migration/seed infrastructure.

Contrast with the Go-based seed pipeline (`business/sdk/dbtest/`): that pipeline
generates randomized test data for unit/integration testing. The customer seed pipeline
provides stable, deterministic data for a named customer's production or demo environment.

---

## Three-Tier Model

### Tier 1: Platform (baseline)

SQL files pg_dumped from the development database after running migrations + the Go seed
pipeline. These are committed as numbered SQL files and contain:

- Geography: countries, regions, cities, streets, timezones
- Core config: currencies, roles, permissions, table access grants
- User approval and HR statuses
- Page/form/table builder configurations (from `seedTableBuilder`, `seedPages`, `seedForms`)
- Workflow infrastructure: action templates, automation rules

Numbering: `00` through `13`. These files are identical across customers and should not
be modified per-customer.

### Tier 2: Customer (real data)

YAML files describing the customer's actual operations. Read by `generate.py` to produce
numbered SQL insert files:

- Their real warehouses and physical layout
- Zones and inventory locations
- Product catalog and suppliers

Numbering: `20` and above (customer-specific range). Files `14`–`19` are reserved.

### Tier 3: Placeholder (generated historical depth)

Synthetic but realistic data generated by `generate.py` using seeded RNG and Faker:

- Virtual employees and organizational structure
- Sample purchase orders spread over months
- Inventory transactions and adjustments to build realistic ledger history
- Order history to populate dashboards and KPI charts

The generator produces these as SQL insert files in the same numbered sequence.

## File Structure

```
deployments/customers/<name>/
├── data/                   # YAML source — customer-supplied real data
│   ├── config.yaml         # Generator settings (name, locale, date ranges)
│   ├── known_ids.yaml      # Stable UUIDs resolved from platform SQL (roles, statuses, etc.)
│   ├── warehouses.yaml     # Physical warehouse locations (addresses, street FK)
│   ├── products.yaml       # Product catalog (names, UPCs, categories, brands)
│   └── suppliers.yaml      # Supplier companies and contacts
├── seed/                   # SQL files applied in order by seed.sh
│   ├── 00_*.sql            # Platform baseline (pg_dumped, committed)
│   │   ...
│   ├── 13_*.sql            # Last platform file (committed)
│   ├── 20_*.sql            # First customer-specific file (generated, gitignored)
│   │   ...
│   └── NN_*.sql            # Additional generated files (gitignored)
├── generate.py             # Reads data/ YAML → writes seed/ SQL
└── seed.sh                 # Runs all seed/ files against a target DB in order
```

## Numbering Convention

| Range | Owner | Contents |
|-------|-------|----------|
| 00–13 | Platform | Geography, config, roles, statuses, page/form/table configs, workflow infra |
| 14–19 | Reserved | — |
| 20+   | Customer | Warehouses, zones, locations, products, suppliers, employees, orders, inventory |

Files execute in lexicographic order. Each file is single-concern and kept under ~500 lines.
Split by domain if a file exceeds that threshold.

## YAML Schemas

### config.yaml

Generator settings and metadata:

- `customer_name` — display name used in generated data
- `locale` — Faker locale (e.g., `en_US`)
- `seed` — RNG seed for reproducibility
- `date_range` — start/end for historical data generation (ISO dates)
- `employee_count` — how many placeholder employees to generate

### known_ids.yaml

Stable UUIDs that the generator needs as foreign keys. These come from the platform SQL
files and are resolved once after running the platform tier:

- Role UUIDs (e.g., `zzzadmin_role_id`)
- User approval status UUIDs (PENDING, APPROVED, etc.)
- Currency UUIDs
- Any status IDs whose names are stable but UUIDs vary per environment

### warehouses.yaml

One entry per warehouse:

- `name`, `address` — display fields
- `street_id` — FK into `geography.streets` (required; must exist in platform data)
- `created_at`, `updated_at` — explicit timestamps for historical placement

### products.yaml

One entry per product:

- `name`, `description`, `upc_code` — display and identification fields
- `brand_id` — FK; brands require `contact_infos_id` (see schema awareness below)
- `product_category_id` — FK
- `status` — must be set (NOT NULL)
- `is_active`, `is_perishable` — boolean flags (NOT NULL)
- `units_per_case` — integer (NOT NULL)
- `tracking_type` — enum value (NOT NULL)

### suppliers.yaml

One entry per supplier:

- `name`, `contact_name`, `email`, `phone`
- `payment_term_id` — FK (payment terms are stable-named in platform data)
- `address` — linked to geography

## Generator (generate.py)

### Deterministic UUIDs

All generated UUIDs use `uuid5` with a fixed namespace and a deterministic string key:

```python
uuid5(NAMESPACE_DNS, f"{customer_name}:{entity_type}:{stable_key}")
```

This guarantees that re-running `generate.py` with the same inputs produces identical SQL,
and that FK references between generated files are stable.

### Seeded RNG

`random.seed(config.seed)` is called once at startup. All random selections (dates,
quantities, relationships between generated entities) derive from this seed. Changing the
seed produces a different but equally valid dataset; keeping the same seed produces
identical output across runs.

### Faker

`faker.Faker(config.locale)` generates realistic placeholder text for names, addresses,
emails, and company names used in placeholder-tier data. Faker is also seeded for
reproducibility.

### Execution flow

1. Load `config.yaml` — settings and date range
2. Load `known_ids.yaml` — platform UUIDs needed as FKs
3. Load `warehouses.yaml`, `products.yaml`, `suppliers.yaml` — customer real data
4. Emit `20_*.sql` through `NN_*.sql` into `seed/`
5. Each SQL file is a single `BEGIN` / `COMMIT` block with INSERT statements for one domain

## Generated Files

Files `20_*.sql` through `27_*.sql` are **gitignored** — they are deterministic output of
`generate.py` and should be regenerated locally rather than committed. This keeps the repo
clean (these files total ~21K lines) and avoids noisy diffs when YAML data changes.

Platform baseline files (`00`–`13`) ARE committed since they're pg_dump snapshots.

## How to Run

From the project root:

```bash
make dev-python               # One-time: install Python + deps
make dev-bounce-manitowoc     # Full cluster up + Manitowoc seed
make reseed-manitowoc         # Recreate DB + migrate + reseed
make seed-manitowoc           # Just seed against existing DB
```

Or run directly:

```bash
./seed.sh <db_url>
```

Runs `generate.py` first, then applies all `seed/*.sql` files against `<db_url>` in filename
order using `psql`.

### Recovery (start_from)

If seeding fails partway through, set `start_from` to resume from a specific file:

```bash
START_FROM=20 ./seed.sh <db_url>
```

Skips files numbered below `start_from`. Useful when platform files are already applied
and only customer files need to re-run.

## AI Readability

Each SQL file is single-concern (one domain, one entity group). Files are kept under ~500
lines. If a domain's inserts exceed ~500 lines, split into multiple numbered files by
sub-domain (e.g., `22_products_catalog.sql`, `23_products_costs.sql`).

This budget ensures any SQL file fits in a single LLM context window and can be reasoned
about without loading adjacent files.

## ⚠ Schema Awareness

These constraints have been found in practice. Violating them produces FK errors at seed
time.

### Warehouses

- Require `street_id` FK → `geography.streets`
- Geography chain must be satisfied in order: region → city → street
- If using a real address, the city and street must be inserted in the platform tier or
  earlier in the customer tier before the warehouse file runs

### Zones

- The `inventory.zones` table has NO `code` column
- Columns: `name`, `description`, `stage` (plus `warehouse_id` FK and audit fields)
- Do NOT include `code` in zone inserts — the column does not exist

### Inventory Locations

- Use coordinate columns: `aisle`, `rack`, `shelf`, `bin`
- There are NO `name` or `code` columns on `inventory.locations`
- Coordinates identify a location; use them for display and FK lookups

### Brands

- Require `contact_infos_id` FK → `core.contact_infos`
- Insert the contact info row before the brand row
- Contact infos require at minimum: `first_name`, `last_name`, `email`

### Products

All of the following columns are NOT NULL and must be explicitly provided:

| Column | Notes |
|--------|-------|
| `description` | Cannot be empty string |
| `upc_code` | Must be unique |
| `status` | Enum value — check current valid values |
| `is_active` | Boolean |
| `is_perishable` | Boolean |
| `units_per_case` | Integer ≥ 1 |
| `tracking_type` | Enum value — check current valid values |

Missing any of these causes a NOT NULL constraint violation at insert time.
