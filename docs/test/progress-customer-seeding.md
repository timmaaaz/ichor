# Progress Summary: customer-seeding.md

## Overview
Customer-specific seed data pipeline outside the Go codebase. Provides real, meaningful data for specific customer deployments without touching core migration/seed infrastructure. Contrast with Go-based test seeding: customer pipeline provides stable, deterministic data for named production/demo environments.

## Three-Tier Model

### Tier 1: Platform (Baseline)

Seeded by two steps that run before customer data:

1. **`make migrate`** — runs `seed.sql` which provides:
   - Geography (countries, regions, timezones)
   - Currencies
   - admin_gopher user
   - ZZZADMIN role
   - Approval statuses
   - Payment terms
   - Table access

2. **`make seed-platform`** — runs Go `InsertPlatformConfig()` function which provides:
   - Page configs
   - Form definitions
   - Table builder configs
   - Workflow templates
   - Alerts

### Tier 2: Customer (Real Data)

YAML files describing customer's actual operations. Read by `generate.py` to produce numbered SQL insert files:
- Real warehouses and physical layout
- Zones and inventory locations
- Product catalog and suppliers

**Numbering:** `20` and above (customer-specific range). Files `14`–`19` reserved.

### Tier 3: Placeholder (Generated Historical Depth)

Synthetic but realistic data generated by `generate.py` using seeded RNG and Faker:
- Virtual employees and organizational structure
- Sample purchase orders spread over months
- Inventory transactions and adjustments (build realistic ledger history)
- Order history (populate dashboards and KPI charts)

## File Structure

```
deployments/customers/<name>/
├── data/                   # YAML source — customer-supplied real data
│   ├── config.yaml         # Generator settings (name, locale, date ranges)
│   ├── known_ids.yaml      # Stable UUIDs from platform SQL
│   ├── warehouses.yaml     # Physical warehouse locations
│   ├── products.yaml       # Product catalog
│   └── suppliers.yaml      # Supplier companies and contacts
├── seed/                   # SQL files applied in order by seed.sh
│   ├── 00_*.sql            # Platform baseline (pg_dumped, committed)
│   ├── 13_*.sql            # Last platform file (committed)
│   ├── 20_*.sql            # First customer-specific (generated, gitignored)
│   └── NN_*.sql            # Additional generated files (gitignored)
├── generate.py             # Reads data/ YAML → writes seed/ SQL
└── seed.sh                 # Runs all seed/ files against target DB
```

## Numbering Convention

| Range | Owner | Contents |
|-------|-------|----------|
| — | Platform (Go) | Seeded by `make migrate` + `make seed-platform` (not SQL files) |
| 20+ | Customer (SQL) | Generated by `generate.py` |

**Ordering:** Files execute in lexicographic order. Single-concern (one domain, one entity group). Kept under ~500 lines. Split by sub-domain if exceeding threshold.

## YAML Schemas

### config.yaml
Generator settings and metadata:
- `customer_name` — display name for generated data
- `locale` — Faker locale (e.g., `en_US`)
- `seed` — RNG seed for reproducibility
- `date_range` — start/end for historical data generation (ISO dates)
- `employee_count` — how many placeholder employees to generate

### known_ids.yaml
Stable UUIDs needed as foreign keys. Resolved from platform SQL:
- Role UUIDs (e.g., `zzzadmin_role_id`)
- User approval status UUIDs (PENDING, APPROVED, etc.)
- Currency UUIDs
- Status IDs with stable names but variable UUIDs per environment

### warehouses.yaml
One entry per warehouse:
- `name`, `address` — display fields
- `street_id` — FK into `geography.streets` (required)
- `created_at`, `updated_at` — explicit timestamps

### products.yaml
One entry per product:
- `name`, `description`, `upc_code` — display and identification
- `brand_id` — FK; brands require `contact_infos_id`
- `product_category_id` — FK
- `status` — must be set (NOT NULL)
- `is_active`, `is_perishable` — boolean flags (NOT NULL)
- `units_per_case` — integer (NOT NULL)
- `tracking_type` — enum value (NOT NULL)

### suppliers.yaml
One entry per supplier:
- `name`, `contact_name`, `email`, `phone`
- `payment_term_id` — FK (stable-named in platform)
- `address` — linked to geography

## Generator (generate.py)

### Deterministic UUIDs
All generated UUIDs use `uuid5` with fixed namespace:

```python
uuid5(NAMESPACE_DNS, f"{customer_name}:{entity_type}:{stable_key}")
```

Guarantees:
- Re-running with same inputs produces identical SQL
- FK references between generated files are stable

### Seeded RNG
`random.seed(config.seed)` called at startup. All random selections (dates, quantities, relationships) derive from this seed:
- Same seed → identical output across runs
- Different seed → different but equally valid dataset

### Faker
`faker.Faker(config.locale)` generates realistic placeholder text for names, addresses, emails, and company names. Also seeded for reproducibility.

### Execution Flow
1. Load `config.yaml` — settings and date range
2. Load `known_ids.yaml` — platform UUIDs needed as FKs
3. Load `warehouses.yaml`, `products.yaml`, `suppliers.yaml` — customer real data
4. Emit `20_*.sql` through `NN_*.sql` into `seed/`
5. Each SQL file is single `BEGIN` / `COMMIT` block with INSERT statements for one domain

## Generated Files

**Gitignored:** Files `20_*.sql` through `27_*.sql` are deterministic output of `generate.py`; should be regenerated locally rather than committed (keeps repo clean; ~21K lines; avoids noisy diffs).

**Committed:** Platform baseline files (`00`–`13`) are pg_dump snapshots.

## How to Run

From project root:

```bash
make dev-python               # One-time: install Python + deps
make dev-bounce-manitowoc     # Full cluster up + Manitowoc seed
make reseed-manitowoc         # Recreate DB + migrate + reseed
make seed-manitowoc           # Just seed against existing DB
```

Or run directly:

```bash
./seed.sh <db_url>
```

Runs `generate.py` first, then applies all `seed/*.sql` files against `<db_url>` in filename order using `psql`.

### Recovery (start_from)

If seeding fails partway through, set `start_from` to resume:

```bash
./seed.sh <db_url> 20
```

Skips files numbered below that value. Useful when platform files already applied and only customer files need to re-run.

## AI Readability

Each SQL file is single-concern (one domain, one entity group). Kept under ~500 lines. If domain inserts exceed ~500 lines, split into multiple numbered files by sub-domain:
- Example: `22_products_catalog.sql`, `23_products_costs.sql`

Ensures any SQL file fits in a single LLM context window and can be reasoned about independently.

## Schema Awareness (Practical Constraints)

### Warehouses
- Require `street_id` FK → `geography.streets`
- Geography chain must be satisfied in order: region → city → street
- If using real address, city and street must be inserted in platform tier or earlier

### Zones
- **NO `code` column** — only `name`, `description`, `stage` (+ `warehouse_id` FK and audit fields)
- Do NOT include `code` in zone inserts

### Inventory Locations
- Use coordinate columns: `aisle`, `rack`, `shelf`, `bin`
- **NO `name` or `code` columns** — coordinates identify location

### Brands
- Require `contact_infos_id` FK → `core.contact_infos`
- Insert contact info row before brand row
- Contact infos require minimum: `first_name`, `last_name`, `email`

### Products
All of these columns are NOT NULL and must be explicitly provided:

| Column | Notes |
|--------|-------|
| `description` | Cannot be empty string |
| `upc_code` | Must be unique |
| `status` | Enum value — check current valid values |
| `is_active` | Boolean |
| `is_perishable` | Boolean |
| `units_per_case` | Integer ≥ 1 |
| `tracking_type` | Enum value — check current valid values |

Missing any causes NOT NULL constraint violation at insert time.

## Critical Points
- **uuid5 determinism** — same inputs produce identical UUIDs across runs
- **Seeded RNG** — changes to seed produce different but equally valid dataset
- **Single-concern files** — each file focuses on one domain for readability
- **Schema constraints** — strictly enforce FK chains (region → city → street → warehouse)
- **NOT NULL columns** — all must be explicitly provided (no defaults for products)

## Notes for Future Development
Customer seeding is well-isolated from test seeding. Most changes will be:
- Adding new YAML schema fields (straightforward, update generator)
- Adjusting historical data generation (moderate, requires RNG testing)
- Changing file numbering scheme (risky, affects existing customer deployments)

The uuid5 determinism is critical — changing it would invalidate existing deployments' FK references.
