== ARCH FILE SUMMARY: customer-seeding.md ==
Date: 2026-03-27

CORE ARCHITECTURE:
- Three-tier customer-specific seed data pipeline (separate from Go test seed infrastructure)
- Lives in deployments/customers/{name}/ (outside main Go codebase)
- Provides stable, deterministic data for named customer deployments
- Contrast: Go-based seed (business/sdk/dbtest/) generates randomized test data

THREE-TIER MODEL:

1. TIER 1: Platform (baseline - seeded first)
   - Step 1: `make migrate` runs seed.sql
     * Geography: countries, regions, timezones
     * Core entities: currencies, admin_gopher user, ZZZADMIN role
     * Platform config: approval statuses, payment terms, table access
   - Step 2: `make seed-platform` runs Go InsertPlatformConfig()
     * Page configs, form definitions, table builder configs
     * Workflow templates, alerts
     * Uses admin_gopher (hardcoded UUID from seed.sql) as created_by
     * Skips all demo data (no fake users, products, orders)

2. TIER 2: Customer (real data - described in YAML)
   - Actual customer operations in deployments/customers/{name}/data/
   - YAML files: config.yaml, known_ids.yaml, warehouses.yaml, products.yaml, suppliers.yaml
   - Read by generate.py to produce SQL insert files
   - Numbering: 20+ (reserved 14-19 for future platform use)
   - Content: real warehouses, zones, locations, product catalog, suppliers

3. TIER 3: Placeholder (synthetic historical depth)
   - Generated by generate.py using seeded RNG + Faker
   - Content: virtual employees, organizational structure
   - Sample purchase orders spread over months
   - Inventory transactions and adjustments (realistic ledger history)
   - Order history for dashboards and KPI charts
   - Numbered files continue from customer tier

DIRECTORY STRUCTURE:

deployments/customers/{name}/
├── data/                    # YAML source (customer-supplied)
│   ├── config.yaml          # Generator settings (name, locale, seed, date_range, employee_count)
│   ├── known_ids.yaml       # Stable UUIDs from platform SQL (roles, statuses, currencies)
│   ├── warehouses.yaml      # Physical warehouse locations (name, address, street_id, timestamps)
│   ├── products.yaml        # Product catalog (name, UPC, category, brand, status flags, tracking)
│   └── suppliers.yaml       # Supplier companies (name, contact, payment_term_id, address)
├── seed/                    # SQL files (applied in order)
│   ├── 00_*.sql to 13_*.sql # Platform baseline (pg_dumped, committed)
│   ├── 20_*.sql to NN_*.sql # Customer+placeholder generated (gitignored)
├── generate.py              # Reads data/ YAML → writes seed/ SQL
└── seed.sh                  # Runs all seed/*.sql files in order

FILE NUMBERING CONVENTION:

Range    | Owner        | Contents
---------|--------------|----------
00-13    | Platform     | Seeded by make migrate + make seed-platform (NOT SQL files, committed if pg_dumped)
14-19    | Reserved     | Future platform extensions
20+      | Customer     | Generated by generate.py (gitignored)

Execution order: lexicographic (00 before 01 before 20 etc)
Design principle: single-concern per file, <500 lines for AI readability

YAML SCHEMAS:

config.yaml:
  - customer_name: display name (used in generated data)
  - locale: Faker locale (e.g., en_US)
  - seed: RNG seed for reproducibility
  - date_range: [start_iso, end_iso] for historical data generation
  - employee_count: number of placeholder employees to generate

known_ids.yaml:
  - Role UUIDs (zzzadmin_role_id, etc)
  - User approval status UUIDs (PENDING, APPROVED)
  - Currency UUIDs
  - Any stable-named entities with variable UUIDs per environment

warehouses.yaml (per warehouse):
  - name, address: display fields
  - street_id: FK → geography.streets (REQUIRED)
  - created_at, updated_at: explicit timestamps

products.yaml (per product):
  - name, description, upc_code: identification
  - brand_id: FK → brands (which requires contact_infos_id)
  - product_category_id: FK → product_categories
  - status: enum (NOT NULL)
  - is_active, is_perishable: booleans (NOT NULL)
  - units_per_case: integer (NOT NULL)
  - tracking_type: enum (NOT NULL)

suppliers.yaml (per supplier):
  - name, contact_name, email, phone: identification
  - payment_term_id: FK → payment_terms
  - address: linked to geography

GENERATOR ALGORITHM (generate.py):

1. Load configuration (config.yaml, known_ids.yaml, all data/*.yaml files)
2. Call random.seed(config.seed) for reproducibility
3. Initialize faker.Faker(config.locale) for realistic placeholder text
4. Generate UUIDs using uuid5:
   uuid5(NAMESPACE_DNS, f"{customer_name}:{entity_type}:{stable_key}")
   → Guarantees identical output on re-run + stable FK references across generated files
5. Emit SQL files 20_*.sql through NN_*.sql into seed/
6. Each SQL file: single BEGIN/COMMIT block, single-concern domain

Key generator properties:
- Deterministic UUIDs (uuid5 from fixed namespace + stable keys)
- Seeded RNG (random.seed) for reproducible data generation
- All random selections (dates, quantities, relationships) from single seed
- Changing seed = different but equally valid dataset
- Same seed = identical output across runs

EXECUTION (from project root):

Development shortcuts:
  make dev-python              # Install Python + dependencies (one-time)
  make dev-bounce-manitowoc    # Full cluster up + Manitowoc seed
  make reseed-manitowoc        # Recreate DB + migrate + reseed
  make seed-manitowoc          # Just seed against existing DB

Direct execution:
  ./seed.sh <db_url>           # Run generate.py + apply all seed/*.sql in order

Recovery (partial failure):
  ./seed.sh <db_url> 20        # Resume from file 20 (skips 00-19)
  Useful: platform files already applied, only customer files need re-run

CRITICAL SCHEMA CONSTRAINTS:

Warehouses:
  - MUST have street_id FK → geography.streets
  - Geography chain: region → city → street (must insert in order)
  - Real addresses: city/street must exist in platform or earlier customer files

Zones:
  - NO code column (common mistake!)
  - Columns: name, description, stage, warehouse_id (+ audit fields)
  - Do NOT include code in zone inserts

Inventory Locations:
  - Use coordinates: aisle, rack, shelf, bin (for identification)
  - NO name or code columns (common mistake!)
  - Coordinates used for display and FK lookups

Brands:
  - MUST have contact_infos_id FK → core.contact_infos
  - Insert contact info before brand row
  - Contact info minimum: first_name, last_name, email

Products (all NOT NULL):
  - description: cannot be empty string
  - upc_code: must be unique
  - status: enum value (check current valid values)
  - is_active: boolean
  - is_perishable: boolean
  - units_per_case: integer ≥ 1
  - tracking_type: enum value (check current valid values)

AI READABILITY DESIGN:

- Single-concern per SQL file (one domain, one entity group)
- File size budget: <500 lines (fits single LLM context)
- If domain exceeds budget, split by sub-domain:
  Example: 22_products_catalog.sql, 23_products_costs.sql
- Purpose: ensure any SQL file can be reasoned about independently

DIFFERENCES FROM GO SEED PIPELINE:

Go pipeline (business/sdk/dbtest/):
  - Generates randomized test data for unit/integration tests
  - Lives in codebase
  - Single run per test execution
  - Not committed to repo

Customer pipeline (deployments/customers/):
  - Provides stable, deterministic customer-specific data
  - Lives outside Go codebase
  - Single run per deployment/environment setup
  - Platform baseline committed, generated files gitignored
  - Supports partial re-seeding (recovery from failure)

VERIFIED FACTS:
- Generated files total ~21K lines across all customer deployments
- Files 20-27 are typically generated per customer
- Known_ids.yaml resolves stable UUIDs from platform SQL
- Seeding requires PostgreSQL with seed.sql structure
- Faker locale affects placeholder text generation (en_US default example)
